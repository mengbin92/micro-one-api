# 第二阶段实施方案

## 1. 阶段定位

第二阶段的目标不是继续扩目录，而是把第一阶段已经建立的 Kratos 骨架推进到“主链路可真实承载流量”的状态。

本阶段聚焦三件事：

1. 迁移用户鉴权与 Token 校验。
2. 迁移渠道选择与渠道快照查询。
3. 将 `relay-gateway` 从调用链验证升级为 OpenAI 兼容 HTTP 网关。

账务扣费、消费流水、后台管理 API 不在本阶段完整迁移范围内，只保留必要的接口接入点。

## 2. 当前基线

第一阶段已经完成：

1. 单仓多服务目录结构。
2. `identity-service`、`channel-service`、`relay-gateway`、`billing-service`、`admin-api` 基础目录。
3. `identity`、`channel`、`billing`、`admin`、`relay` proto 契约。
4. `identity-service` 与 `channel-service` 的基础 gRPC 服务。
5. `relay-gateway -> identity-service -> channel-service` 的最小 gRPC 调用链。
6. 内存数据源与 MySQL DSN 数据源的基础适配。

第二阶段应基于这些成果继续，不重新设计服务边界。

## 3. 阶段目标

满足以下条件即可认为第二阶段完成：

1. `identity-service` 能基于现有 one-api 表完成真实 Token 鉴权。
2. `channel-service` 能基于现有 one-api 表完成真实渠道选择。
3. `relay-gateway` 暴露 OpenAI 兼容的最小 HTTP 接口。
4. 请求进入 `relay-gateway` 后，必须经过鉴权、模型权限校验、渠道选择，再转发到上游。
5. 本阶段不直接修改用户余额和 Token 余额，只预留 billing 接口。
6. 本阶段主链路具备可重复的集成验证方式。

## 4. 非目标

本阶段暂不做：

1. 完整账务预扣、提交、释放。
2. 完整消费日志迁移。
3. 后台管理 API 迁移。
4. 多租户、组织、项目等新概念引入。
5. 独立分库。
6. 复杂服务发现、配置中心和链路追踪平台接入。

这些能力应在主链路稳定后逐步引入。

## 5. 服务改造范围

### 5.1 `identity-service`

本阶段负责：

1. 兼容 one-api 现有 `users`、`tokens` 表。
2. 校验 Token 是否存在。
3. 校验 Token 状态。
4. 校验 Token 过期时间。
5. 校验 Token 剩余额度或无限额度标记。
6. 校验用户状态。
7. 输出 relay 所需的鉴权快照。
8. 输出 Token 可用模型列表。

本阶段应补齐：

1. 鉴权错误到 gRPC/HTTP 错误码的统一映射。
2. Token 前缀和 Authorization Header 解析规则。
3. 用户分组默认值处理。
4. Token 模型白名单为空时的语义定义。
5. `identity` usecase 单元测试。

### 5.2 `channel-service`

本阶段负责：

1. 兼容 one-api 现有 `channels`、`abilities` 表。
2. 按 `group + model` 查询可用渠道。
3. 过滤禁用渠道。
4. 按优先级选路。
5. 支持同优先级随机选择。
6. 返回 relay 所需的渠道快照。

本阶段应补齐：

1. 渠道类型到 provider adapter 的映射规则。
2. 渠道 base URL 默认值处理。
3. 渠道密钥脱敏日志策略。
4. `exclude_first_priority` 的清晰语义和测试。
5. `channel` usecase 单元测试。

### 5.3 `relay-gateway`

本阶段负责：

1. 启动 Kratos HTTP 服务。
2. 暴露 OpenAI 兼容最小接口。
3. 从 Authorization Header 解析 Token。
4. 解析请求模型。
5. 调用 `identity-service` 获取鉴权快照。
6. 校验模型是否允许。
7. 调用 `channel-service` 获取渠道。
8. 根据渠道类型调用对应 provider adapter。
9. 将上游响应按 OpenAI 兼容格式返回。

本阶段建议先支持：

1. `POST /v1/chat/completions`
2. `GET /v1/models`

`/v1/completions`、图片、音频、rerank、embeddings 可后置。

### 5.4 `billing-service`

本阶段仅做接入点准备：

1. 保持现有 proto。
2. 在 `relay-gateway` 中预留调用位置。
3. 不在主链路中强制启用扣费。

如果要在第二阶段提前验证账务，只允许以 mock 或 no-op billing client 方式接入，不能把扣费逻辑散落回 `relay-gateway`。

### 5.5 `admin-api`

本阶段不迁移后台业务。

允许补充：

1. 健康检查。
2. 服务启动骨架。
3. 后续管理 API 路由占位。

## 6. OpenAI 兼容接口范围

### 6.1 `POST /v1/chat/completions`

本阶段支持最小请求字段：

1. `model`
2. `messages`
3. `stream`
4. `temperature`
5. `max_tokens`

请求处理流程：

1. 解析 Bearer Token。
2. 解析 `model`。
3. 调用 `identity-service`。
4. 校验 Token 是否允许访问该模型。
5. 调用 `channel-service`。
6. 构造上游请求。
7. 转发到上游。
8. 返回上游响应。

流式响应可以分两步落地：

1. 第一小步先支持非流式。
2. 第二小步支持 `stream=true` 的 SSE 透传。

### 6.2 `GET /v1/models`

本阶段支持：

1. 解析 Token。
2. 获取用户分组。
3. 调用 `channel-service` 查询可用模型。
4. 与 Token 模型白名单取交集。
5. 返回 OpenAI 兼容 models 列表。

## 7. 鉴权与模型权限规则

### 7.1 Token 来源

本阶段只接受：

```text
Authorization: Bearer <token>
```

暂不支持 query 参数传 Token，避免新链路继续扩大不安全入口。

### 7.2 模型权限

规则：

1. Token 模型白名单为空时，表示不额外限制模型。
2. Token 模型白名单非空时，请求模型必须在白名单内。
3. 最终可用模型仍必须由 `channel-service` 确认存在可用渠道。

### 7.3 错误码

建议映射：

1. Token 缺失：`401`
2. Token 无效：`401`
3. Token 过期：`401`
4. Token 额度不足：`429`
5. 用户禁用：`403`
6. 模型无权限：`403`
7. 无可用渠道：`503`
8. 上游失败：透传上游状态码；无法识别时返回 `502`

## 8. Provider Adapter 范围

第二阶段只实现 OpenAI-compatible adapter。

职责：

1. 接收 relay 的标准化请求。
2. 应用渠道 `base_url` 和 `key`。
3. 构造上游 HTTP 请求。
4. 透传响应头中必要字段。
5. 支持非流式响应。
6. 后续补充 SSE 透传。

暂不做：

1. 各 provider 差异化参数翻译。
2. Anthropic 原生协议转换。
3. Gemini 原生协议转换。
4. 多模态复杂字段映射。

这些适合在主链路稳定后，按 provider 单独推进。

## 9. 数据兼容策略

本阶段优先兼容现有 one-api 表，不先设计新库。

要求：

1. 读 `users`、`tokens`、`channels`、`abilities` 时必须通过 repo 接口。
2. 不在 service 或 server 层直接访问数据库。
3. 不在 `relay-gateway` 中读取业务表。
4. 字段语义不清时，先在 repo 层显式封装转换函数。
5. 所有兼容性假设写入测试或文档。

如果发现现有表字段不足，只允许新增非破坏性字段或补充派生查询，不在第二阶段做大规模表结构重写。

## 10. 配置策略

本阶段先使用环境变量，后续再统一配置文件加载。

建议保留：

1. `IDENTITY_GRPC_ADDR`
2. `CHANNEL_GRPC_ADDR`
3. `RELAY_HTTP_ADDR`
4. `IDENTITY_GRPC_ENDPOINT`
5. `CHANNEL_GRPC_ENDPOINT`
6. `IDENTITY_SQL_DSN`
7. `CHANNEL_SQL_DSN`
8. `SQL_DSN`

新增建议：

1. `RELAY_PROVIDER_TIMEOUT`
2. `RELAY_DEFAULT_OPENAI_BASE_URL`
3. `RELAY_ENABLE_BILLING`

## 11. 测试策略

### 11.1 单元测试

必须补：

1. `identity` Token 状态测试。
2. `identity` 用户状态测试。
3. `identity` 模型白名单测试。
4. `channel` 优先级选路测试。
5. `channel` 禁用渠道过滤测试。
6. `relay` 模型权限测试。

### 11.2 集成测试

建议补一个最小集成测试：

1. 启动内存版 `identity-service`。
2. 启动内存版 `channel-service`。
3. 启动 `relay-gateway`。
4. 使用 mock OpenAI-compatible upstream。
5. 请求 `/v1/chat/completions`。
6. 验证 relay 返回上游响应。

### 11.3 手工验收

至少保留以下命令级验收路径：

1. 启动 `identity-service`。
2. 启动 `channel-service`。
3. 启动 mock upstream。
4. 启动 `relay-gateway`。
5. curl 调用 `/v1/models`。
6. curl 调用 `/v1/chat/completions`。

## 12. 实施顺序

建议按以下顺序推进：

1. 补 `identity-service` 领域测试和错误映射。
2. 补 `channel-service` 领域测试和选路语义。
3. 抽象 relay provider adapter。
4. 实现 relay HTTP server。
5. 实现 `/v1/models`。
6. 实现 `/v1/chat/completions` 非流式。
7. 接入 mock upstream 做集成验证。
8. 补 `stream=true` SSE 透传。
9. 整理启动脚本或 Makefile。
10. 更新 README 与验收说明。

## 13. 里程碑

### 13.1 M1：鉴权和渠道真实化

产出：

1. `identity` 和 `channel` usecase 测试。
2. one-api 表兼容说明。
3. 错误码映射。

验收：

1. 有效 Token 可以生成鉴权快照。
2. 禁用、过期、额度不足 Token 会被拒绝。
3. 指定 group 和 model 能选出可用渠道。

### 13.2 M2：Relay HTTP 网关

产出：

1. Kratos HTTP server。
2. `/v1/models`。
3. `/v1/chat/completions` 非流式。
4. OpenAI-compatible adapter。

验收：

1. 无 Token 请求返回 `401`。
2. 无权限模型返回 `403`。
3. 无渠道模型返回 `503`。
4. 有效请求可以转发到 mock upstream。

### 13.3 M3：主链路验收

产出：

1. 集成测试或可重复手工验收脚本。
2. README 启动说明。
3. 第二阶段完成状态说明。

验收：

1. `go test ./...` 通过。
2. 本地三服务加 mock upstream 可跑通。
3. relay 不直接访问用户、Token、渠道业务表。
4. 扣费逻辑没有散落到 relay。

## 14. 风险与控制

### 14.1 风险：过早迁账务

控制：

1. 第二阶段只预留 billing client。
2. 真实扣费放到第三阶段。

### 14.2 风险：Relay 重新变成单体

控制：

1. Relay 只做协议接入、编排和转发。
2. 鉴权归 `identity-service`。
3. 渠道归 `channel-service`。
4. 扣费归 `billing-service`。

### 14.3 风险：Provider 差异拖慢主链路

控制：

1. 第二阶段只做 OpenAI-compatible adapter。
2. 其他 provider 后续逐个迁移。

### 14.4 风险：兼容 one-api 表导致边界污染

控制：

1. 兼容逻辑只放 repo 层。
2. biz 层使用稳定领域模型。
3. service 层只做协议转换。

## 15. 第二阶段完成标准

满足以下条件即可进入第三阶段：

1. `identity-service` 真实鉴权可用。
2. `channel-service` 真实渠道选择可用。
3. `relay-gateway` OpenAI 兼容 HTTP 主链路可用。
4. `/v1/models` 和 `/v1/chat/completions` 可本地验收。
5. 有 mock upstream 集成验证。
6. 主要 usecase 有单元测试覆盖。
7. billing 只作为明确接口边界存在，没有被绕过或散落实现。
