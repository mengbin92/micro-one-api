# 第一阶段实施方案

## 1. 阶段目标

第一阶段不是把 `one-api` 一次性全部迁完，而是建立一个能跑通的 Kratos 微服务基础骨架，并完成最小业务闭环设计。

本阶段目标只有 4 个：

1. 确定首批服务边界。
2. 确定仓库结构与 proto 契约。
3. 搭建 Kratos 单仓多服务工程骨架。
4. 明确首个可落地闭环：`relay-gateway -> identity-service -> channel-service`。

本阶段不追求：

1. 一次完成账务迁移。
2. 一次完成管理后台迁移。
3. 一次完成日志、监控、通知全链路迁移。

## 2. 第一阶段服务范围

### 2.1 本阶段实际创建的服务

第一阶段先创建 5 个服务目录，但落地顺序不同：

1. `identity-service`
2. `channel-service`
3. `relay-gateway`
4. `billing-service`
5. `admin-api`

### 2.2 本阶段优先级

优先级按下面顺序推进：

1. `identity-service`
2. `channel-service`
3. `relay-gateway`
4. `billing-service`
5. `admin-api`

原因：

1. `relay-gateway` 要独立，前置条件是鉴权和渠道选择服务先稳定。
2. `billing-service` 设计复杂，第一阶段先定接口和表设计，不要求全部业务完成。
3. `admin-api` 是后台聚合层，不是第一条关键链路。

## 3. 第一阶段交付物

### 3.1 文档交付物

1. 微服务总体改造方案。
2. 基于 Kratos 的落地方案。
3. 第一阶段实施方案。

### 3.2 代码交付物

1. 单仓多服务目录骨架。
2. `api/` 下 proto 草案。
3. 每个服务的 `cmd/configs/internal` 基础目录。
4. 公共 `internal/pkg/` 目录骨架。
5. `third_party/validate`、`third_party/google` 依赖目录。

### 3.3 非目标

第一阶段不要求：

1. 所有 proto 都已经完全稳定。
2. 所有服务都能启动生产可用实例。
3. 所有数据库表都已经重构完成。

## 5. 服务边界

## 5.1 `identity-service`

本阶段负责：

1. 用户查询。
2. Token 校验。
3. 鉴权快照输出。
4. 用户状态、分组、可用模型基础校验。

本阶段先不负责：

1. 完整 OAuth 流程迁移。
2. 所有后台用户管理功能。

### 对应 `one-api` 现有逻辑

主要来源：

1. `controller/user.go`
2. `controller/auth/*.go`
3. `model/user.go`
4. `model/token.go`
5. `middleware/auth.go`

## 5.2 `channel-service`

本阶段负责：

1. 渠道查询。
2. 能力矩阵查询。
3. 按 `group + model` 选路。
4. 返回渠道快照。

本阶段先不负责：

1. 全量后台管理能力。
2. 渠道自动探测和自动恢复。

### 对应 `one-api` 现有逻辑

主要来源：

1. `model/channel.go`
2. `model/ability.go`
3. `model/cache.go`
4. `middleware/distributor.go`

## 5.3 `relay-gateway`

本阶段负责：

1. 提供 `/v1/*` 兼容 API。
2. 调用 `identity-service` 完成鉴权。
3. 调用 `channel-service` 获取路由结果。
4. 预留 `billing-service` 接口接入点。
5. 保留上游 provider 适配层抽象。

本阶段先不负责：

1. 完整账务落地。
2. 全量 provider 完整迁移。

### 对应 `one-api` 现有逻辑

主要来源：

1. `router/relay.go`
2. `controller/relay.go`
3. `relay/`
4. `middleware/distributor.go`

## 5.4 `billing-service`

本阶段负责：

1. 先定义 proto。
2. 先定义表结构方向。
3. 先确定预扣、提交、释放接口。

本阶段允许：

1. 先不实现全量业务。
2. 先不替换 `one-api` 原有全部额度逻辑。

## 5.5 `admin-api`

本阶段负责：

1. 仅作为未来后台聚合层的目录和 proto 占位。

## 6. 第一阶段 proto 范围

## 6.1 `identity/v1/identity.proto`

本阶段必须定义：

1. `ValidateToken`
2. `GetAuthSnapshot`
3. `GetUser`

建议消息：

1. `TokenInfo`
2. `UserInfo`
3. `AuthSnapshot`

## 6.2 `channel/v1/channel.proto`

本阶段必须定义：

1. `SelectChannel`
2. `GetChannel`
3. `ListAvailableModels`

建议消息：

1. `ChannelInfo`
2. `ChannelConfig`
3. `SelectChannelRequest`
4. `SelectChannelReply`

## 6.3 `billing/v1/billing.proto`

本阶段必须定义但可先不全实现：

1. `ReserveQuota`
2. `CommitQuota`
3. `ReleaseQuota`
4. `GetAccountSnapshot`

## 6.4 `admin/v1/admin.proto`

本阶段仅做占位：

1. 用户管理接口预留。
2. 渠道管理接口预留。
3. 配置管理接口预留。

## 6.5 `relay/v1/relay.proto`

本阶段可选：

1. 主要用于内部结构定义，不一定马上对外暴露 gRPC。

## 7. 公共包边界

第一阶段必须严格控制 `internal/pkg/`，避免重新形成“大 common 包”。

### 允许放进 `internal/pkg/` 的内容

1. 统一错误码定义。
2. HTTP/gRPC 中间件。
3. 链路追踪封装。
4. 元数据常量。
5. 鉴权 token 解析基础能力。

### 不允许放进 `internal/pkg/` 的内容

1. 用户业务逻辑。
2. 渠道路由逻辑。
3. 配额业务逻辑。
4. 任何强领域代码。

## 8. 数据库与表归属

第一阶段仍允许共享同一个 MySQL 实例，但从代码归属上就要拆开。

### `identity-service`

归属表：

1. `users`
2. `tokens`

### `channel-service`

归属表：

1. `channels`
2. `abilities`

### `billing-service`

短期涉及：

1. `redemptions`
2. 用户额度字段
3. Token 额度字段

中期新增：

1. `accounts`
2. `quota_reservations`
3. `ledger_entries`

### `config-service`

暂不建服务，但未来归属：

1. `options`

### `log-service`

暂不建服务，但未来归属：

1. `logs`

## 9. 第一阶段最小闭环

第一阶段真正要跑通的是这个链路：

1. 客户端请求 `relay-gateway`
2. `relay-gateway` 调 `identity-service.ValidateToken`
3. `relay-gateway` 调 `identity-service.GetAuthSnapshot`
4. `relay-gateway` 调 `channel-service.SelectChannel`
5. `relay-gateway` 拿到渠道快照后执行本地 relay 编排

这条链路跑通之后，再接：

1. `billing-service`
2. `admin-api`
3. 配置服务

## 10. 实施顺序

### Step 1

生成目录骨架。

### Step 2

落 proto 草案。

### Step 3

为 `identity-service`、`channel-service`、`relay-gateway` 建立基础 Kratos 结构：

1. `cmd`
2. `configs`
3. `internal/biz`
4. `internal/data`
5. `internal/service`
6. `internal/server`

### Step 4

补共享中间件和错误码目录。

### Step 5

开始迁移鉴权快照与渠道选路逻辑。

## 11. 风险控制

### 风险 1

proto 设计反复变动。

应对：

1. 第一阶段只定义最小闭环接口。

### 风险 2

共享包膨胀。

应对：

1. 只放基础设施能力，不放业务逻辑。

### 风险 3

`billing-service` 过早实现导致返工。

应对：

1. 第一阶段只定接口和表方向，业务迁移后置。

## 12. 第一阶段完成标准

满足以下条件即可认为第一阶段基础工作完成：

1. 仓库目录结构稳定。
2. 首批 proto 已落地。
3. 首批 5 个服务目录已建立。
4. `identity-service`、`channel-service`、`relay-gateway` 基础 Kratos 代码骨架已具备。
5. 团队可以基于该骨架继续迁移具体业务逻辑。
